# SlotDesk

SlotDesk - учебный планировщик онлайн-записей на услуги. Клиент оставляет заявку без регистрации, видит ближайшие занятые слоты и не получает доступ к чужим данным. Администратор входит через Supabase Auth, просматривает заявки, меняет статусы, отмечает подготовку SMS и удаляет записи.

## Стек

- React
- TypeScript
- Vite
- Supabase Auth
- Supabase Database / PostgreSQL
- Row Level Security

## Возможности

- публичная форма онлайн-записи без аккаунта;
- выбор услуги, специалиста, даты, времени и длительности;
- проверка номера телефона по коду страны;
- проверка email, даты и времени;
- запрет записи на прошедшую дату;
- запрет двух активных записей на одно время к одному специалисту;
- публичный список занятых слотов;
- вход администратора через Supabase Auth;
- проверка администратора через таблицу `app_admin`;
- поиск и фильтрация заявок в админской части;
- смена статуса записи;
- отметка SMS как отправленного;
- удаление записи администратором.

## Быстрый запуск

```bash
npm install
```

В проекте уже есть `.env.example` с публичными ключами Supabase для проверки работы приложения. Для локального запуска достаточно создать `.env` на его основе:

```bash
cp .env.example .env
```

На Windows можно просто скопировать файл `.env.example` и переименовать копию в `.env`.

После этого:

```bash
npm run dev
```

Если значения в `.env` менялись, dev-сервер нужно полностью перезапустить.

## Переменные окружения

Используются переменные:

```env
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
VITE_SUPABASE_PUBLISHABLE_KEY=
```

Для работы достаточно `VITE_SUPABASE_URL` и одного публичного ключа: `VITE_SUPABASE_ANON_KEY` или `VITE_SUPABASE_PUBLISHABLE_KEY`. В `.env.example` оставлены публичные ключи тестового Supabase-проекта, чтобы проверяющий мог запустить приложение без отдельной настройки базы данных.

В Vercel эти же значения нужно добавить в `Project Settings -> Environment Variables`.

## Архитектура

Проект разложен по FSD-подходу:

- `src/app` - подключение основной страницы приложения;
- `src/pages/scheduler` - экран планировщика и состояние страницы;
- `src/widgets` - крупные блоки интерфейса: шапка, публичное расписание, статистика, список заявок;
- `src/features` - пользовательские действия: создание записи, фильтрация, вход администратора;
- `src/entities` - сущности записи и админской сессии, типы, селекторы, валидация, репозитории;
- `src/shared` - общий клиент Supabase REST и базовая настройка Supabase.

Публичная запись работает через REST-запрос с ролью `anon` и `Prefer: return=minimal`. Клиент может создать запись, но не получает строку из таблицы `appointments`, поэтому публичный `SELECT` на личные данные не нужен. Админская часть работает через токен Supabase Auth.

## Supabase

Схема базы находится в `supabase/schema.sql`. Она создает:

- таблицу `appointments`;
- таблицу `app_admin`;
- представление `appointment_slots`;
- RLS-политики для публичной записи;
- RLS-политики для админского чтения, изменения и удаления;
- уникальный индекс, который не дает создать две активные записи к одному специалисту на одно время.

Если используется приложенный Supabase-проект из `.env.example`, повторно создавать базу и администратора не нужно. Если нужен свой Supabase-проект, надо выполнить `supabase/schema.sql` в SQL Editor и заменить переменные окружения.

## Администратор

В Supabase-проекте из `.env.example` администратор уже создан и добавлен в таблицу `app_admin`. Обычная регистрация пользователей в интерфейсе не используется: клиентская запись остается публичной, а управление заявками доступно только администратору.

Если проект разворачивается на собственной базе, администратора нужно добавить вручную:

1. Открыть `Authentication -> Users`.
2. Создать пользователя с email и паролем.
3. Скопировать `id` пользователя.
4. Выполнить запрос из конца `supabase/schema.sql`, подставив этот `id` в `app_admin`.
